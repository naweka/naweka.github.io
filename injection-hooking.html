Без труда не выловишь и данные с софта... Или выловишь?

>

Постановка проблемы

В жизни каждого ревёрсера наступает момент, когда в руки попадает бинарник, который был обфусцирован инопланетными технологиями какой-то непонятно штукой. И тогда перед ним встаёт тяжёлый моральный выбор: стоит ли это того, чтобы потратить пару недель своей жизни?

С одной стороны, это же увлекательное (нет) приключение, в ходе которого можно ещё немного набить руку, но, с другой стороны, это ж, блин, надо потратить кучу времени, а время (и желание) есть не всегда и не у всех.


Так что же делать?

>

Свет в конце туннеля

На самом деле, выход из такой ситуации есть, причем его можно применять по-разному. А ещё им часто пользуются вирусописатели и любители поиграть в игры с читами. Короче говоря, это DLL-инъекция.

💡DLL-инъекция – это техника внедрения кода в процесс другой программы, позволяющая получить контроль над процессом, с целью повыполнять там всякие штуки или вытащить нужную информацию.

Например, мы можем отследить выполнение какой-то функции, что ей приходит, что выходит. Также можем изменить поведение (ну это уже прям совсем обычный хук). Получается вроде дебаггера на минималках.

Ладно, надушнили достаточно, переходим от слов к делу! Сначала разберёмся с тем, как это всё вообще работает.

>

А что внедрять?

Не будем изобретать велосипед, и в качестве основы возьмём библиотеку Harmony. Она позволяет делать простые префикс- (перед основным методом), вместо- (подмена метода) и постфиксовые (после основного метода) хуки.

Эта либа использует динамические методы и подменяет инструкции оригинального метода на джамп, то есть представляет из себя хук из семейства обыкновенных и вполне себе детектируемых. Но сейчас нас это не особо волнует, главное, что пользоваться ей крайне просто и удобно, за это все её и любят.

Из доступного мы можем сделать как минимум две вещи: хукнуть весь метод 9784C5ED65974E8485BB1D49B7D8E590 постфиксным хуком, чтобы прочесть содержимое или же просто хукнуть Encoding.BigEndianUnicode.GetString тем же постфиксом. Я решил выбрать второй метод, потому что хукнуть метод, который «из коробки», проще.

Код библиотеки получился небольшим:

📝 Код

 [С#]

 
public static class CodePatchExample
{
    static Harmony harmony = new Harmony("tutorial");
    const string logFilePath = "data.txt";

    static void Log(string str)
    {
        File.AppendAllText(
           logFilePath, $"[{DateTime.Now:HH:mm:ss.fff}] {str}\n");
    }

    public static int InstallHooks(string args)
    {
        Log("Hooking...");

        var targetMethod = typeof(Encoding)
            .GetMethod("GetString", new[] { typeof(byte[]) });

        var postfixMethod = new HarmonyMethod(
            typeof(CodePatchExample).GetMethod("PostHook"));

        harmony.Patch(targetMethod, postfix: postfixMethod);

        return 0;
    }

    // Encoding.GetString возвращает string, его мы и хотим получить
    public static void PostHook(string __result)
    {
        Log(__result);
    }
}

Отлично, получилось не очень сложно, можно перейти к инжектору.

А как внедрять?

В теории, мы могли бы взять какой-нибудь стандартный инжектор и радоваться жизни. Что-то вроде Process Hacker‘а или что-то ещё… Но тут стоит напомнить, что честь имеем с dotNet. А этот dotNet – птица гордая, пока не пнёшь – не полетит. Для внедрения управляемой DLL в процесс необходимо наличие в нём CLR среды. Поэтому не всё так просто, придется писать свой CLR Hosting, потом его инжектить… Ну или взять готовый Managed Injector.

Managed Injector позволит нам сделать магию и внедрить одну управляемую сборку в процесс с другой. В readme написано, как его использовать, я буду использовать консольный интерфейс. Можно, конечно, пользоваться им прям как библиотекой, но я не хочу зависеть от разрядности самого загрузчика, проще прописать выбор между двумя версиями консольной утилиты для 32/64 бит.

Кстати, разрядность можно проверить вот таким трюком:

📝 Код

 [С#]

 
static byte[] ReadBytes(string filePath, int offset, int length)
{
    using (FileStream fs = new FileStream(filePath, FileMode.Open, FileAccess.Read))
    {
        fs.Seek(offset, SeekOrigin.Begin);
        byte[] bytes = new byte[length];
        fs.Read(bytes, 0, length);
        return bytes;
    }
}

FileInfo fileInfo = new FileInfo(config.FilePath);
bool is64Bit = BitConverter.ToUInt16(ReadBytes(fileInfo.FullName, 0x3C, 2), 0) == 0x20B;

Этот трюк основан на том факте, что по смещению 0x3C у 32-х битных бинарей лежит 0x10B, а у 64-х битных – 0x20B.

Вернёмся к обсуждению нашего плана по внедрению DLL’ки. Это не обязательно, но я хотел сделать инжект при первой же возможности (чтобы минимизировать количество неконтролируемых инструкций), но в моём случае это не так просто.

Если мы имеем дело с обычным нативным бинарником, то достаточно всего лишь с помощью CreateProcess запускать процесс сразу же замороженным. Однако среда CLR не доступна сразу же при создании процесса, а ведь она нужна для корректной работы инжектора.

Тогда напишем метод, который умеет терпеливо ждать, пока среда CLR в процессе не станет доступной. К счастью, это не очень сложно, достаточно мониторить загруженные модули процесса:

📝 Код

 [С#]

 
public static bool ClrIsLoaded(int pID)
{
    ProcessModuleCollection pModuleCollection = Process.GetProcessById(pID).Modules;
    for (int i = 0; i < pModuleCollection.Count; i++)
    {
        if (pModuleCollection[i].ModuleName.ToLower() == "clr.dll")
            return true;
    }
    return false;
}

Также нужно заморозить процесс, чтобы как можно меньше кода выполнялось вне нашего «шпиона»:

📝 Код

 [С#]

 
[DllImport("kernel32.dll")]
private static extern IntPtr OpenThread(
    int dwDesiredAccess, 
    bool bInheritHandle, 
    uint dwThreadId);

[DllImport("kernel32.dll")]
private static extern uint SuspendThread(IntPtr hThread);

[DllImport("kernel32.dll")]
private static extern uint ResumeThread(IntPtr hThread);

[DllImport("kernel32", CharSet = CharSet.Auto, SetLastError = true)]
private static extern bool CloseHandle(IntPtr handle);

public static void SuspendProcess(int pid)
{
    var process = Process.GetProcessById(pid);
    foreach (ProcessThread pThread in process.Threads)
    {
        IntPtr pOpenThread = OpenThread(0x2, false, (uint)pThread.Id);
        SuspendThread(pOpenThread);
        CloseHandle(pOpenThread);
    }
}

public static void ResumeProcess(int pid)
{
    var process = Process.GetProcessById(pid);
    foreach (ProcessThread pThread in process.Threads)
    {
        IntPtr pOpenThread = OpenThread(0x2, false, (uint)pThread.Id);
        ResumeThread(pOpenThread);
        CloseHandle(pOpenThread);
    }
}

Впринципе, это всё, что нужно. План надёжен, как швейцарские часы: запускаем процесс, ждём, когда будет доступен CLR, потом замораживаем процесс, инжектим DLL’ку, размораживаем процесс.

Погнали смотреть, что получилось!
